Topology

Here are a few questions about the underlying connectivity of a hypertext web.

Two- or Multi-ended Links

The term "link" normally indicates two ends. Variations include multiple sources/destinations, or constructs relating more than two anchors. This maps onto logic systems and predicate calculus (see "Aquanet" system from Xerox PARC - HT91). For simplicity, we focus on conventional two-ended links.

Monodirectional or Bidirectional Links

Bidirectional links exist in both directions. Pros: users may see the reverse relationship (e.g., if A includes B, B is part of A). Cons: may constrain authors. Examples: KMS & HyperCard have one-way links; Enquire has two-way links. Useful for protected databases and managing dangling links. See Building Back-links.

Anchors with Multiple Links

One anchor may lead to multiple links, or a link may have multiple anchors. Many-to-one mapping is simpler. JFG suggests showing a small menu for ambiguous links. Example system: Microcosm.

Typed Links

Typed links carry semantic information, helping the system manage data. Default type exists if untyped. See also list of some types. JFG and TBL debate whether links can have multiple types. Typed links help with graphical overviews and automatic tracing.

Ancillary Information on Links

Links may include date, version, authorship, or comments. Best practice: store information in source/intermediate nodes instead of overloading the link itself. Links can be annotatable but avoid complicating the interface.

Preview Information

Preview information allows readers to see link content before following it. Better approach: caching target node titles or descriptive material. Fast graphical overviews can replace previews.
